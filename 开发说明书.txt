# tangerine 桌面 agent框架 - 开发说明书

## 一、技术栈与核心依赖

- **桌面壳层**：Tauri 2（Rust）
- **前端**：React 18 + TypeScript + Vite 6 + Tailwind CSS v4
- **工作流画布**：@xyflow/react
- **Markdown 渲染**：
  - react-markdown
  - remark-gfm
  - rehype-highlight
  - highlight.js
- **Tauri 插件**：
  - @tauri-apps/plugin-shell
  - @tauri-apps/plugin-dialog
  - @tauri-apps/plugin-fs
- **Python Sidecar**：FastAPI
- **Rust 图片处理**：base64

## 三、架构设计与解耦说明

### 1. 聊天核心状态解耦 (`useChatManager`)
为避免单一 Hook 文件过于臃肿导致维护困难和性能瓶颈，`useChatManager` 采用了**组合式自定义 Hook** 的设计模式，按职责拆分为 5 个子模块：
- **`useChatUIState`**：纯粹管理 UI 交互状态（如输入框文本、菜单显隐、拖拽状态），隔离 DOM 事件监听。
- **`useModelManager`**：专注处理 API 提供商配置解析、模型列表同步以及基于角色的模型推断。
- **`useAttachmentManager`**：封装 Tauri 原生文件选择器、拖放事件监听以及 Rust 侧图片 Base64 预览生成。
- **`useSessionManager`**：管理 LocalStorage 持久化、会话树操作（包含复杂的分支切换与深度删除防复活逻辑）。
- **`useStreamChat`**：处理与 Python Sidecar 的 SSE 流式通信、数据解析、中断控制（AbortController）及错误降级。

**优势**：主入口 `useChatManager.ts` 仅负责依赖注入和跨模块副作用联动，对外暴露的 `state`、`refs` 和 `actions` 接口保持 100% 兼容，调用方（如 `App.tsx`）无需感知内部重构。

### 2. 自动化画布核心状态解耦 (`FlowEditor`)
工作流画布 (`FlowEditor.tsx`) 包含大量复杂的交互逻辑（如拖拽创建、集合节点嵌套计算、剪贴板坐标映射等）。为避免“上帝组件”问题，将其核心逻辑剥离至 `automation/hooks/` 目录下的 6 个独立 Hook 中：
- **`useFlowHistory`**：核心状态中枢。管理 `nodes` 和 `edges` 状态，处理历史记录栈（撤销/重做），以及节点拖拽、缩放时的快照记录逻辑。
- **`useFlowPersistence`**：IO 副作用隔离。专门负责从 LocalStorage 加载画布数据，并在状态变化时进行防抖保存，同时处理会话切换时的状态重置。
- **`useFlowRoles`**：数据源管理。负责角色数据的加载、监听 `storage` 事件实现跨窗口同步，并将角色数据映射为画布可用的模板列表。
- **`useFlowTemplateDrag`**：交互逻辑封装。处理从工具栏拖拽模板到画布的全局指针事件，并计算屏幕坐标到画布坐标的转换以创建新节点。
- **`useFlowClipboard`**：复杂业务逻辑隔离。专门处理复制、粘贴（包含集合节点子孙节点的复杂坐标计算与 ID 映射）以及删除逻辑。
- **`useFlowShortcuts`**：全局事件绑定。纯粹负责监听全局键盘事件，并分发到对应的操作函数。

**优势**：重构后的 `FlowEditor.tsx` 代码量大幅减少，从“业务逻辑处理中心”转变为纯粹的“状态组装器”和“视图渲染层”。这不仅降低了认知负担，还通过物理隔离减少了无关状态更新导致的闭包重建，提升了大型画布的渲染性能。

### 3. 自动化节点组件与工具函数解耦 (`AutomationNodes` & `NodeUtils`)
为避免单文件代码量过大导致的维护困难与渲染性能瓶颈，对节点视图层与工具函数层进行了深度解耦与重构：
- **节点组件物理拆分**：将原本臃肿的 `AutomationNodes.tsx` 拆分为 `nodes/` 子目录。提取了 `WorkflowNodeShell` (外壳) 和 `NodeSelect` (下拉框) 作为基础 UI 组件，并将 7 种具体节点 (Role, Tool, Trigger 等) 拆分为独立文件。主文件仅作为 `nodeTypes` 的注册导出入口。
- **工具函数配置驱动与算法优化**：重构 `automationNodeUtils.ts`，将长达百行的 `if-else` 节点工厂函数重构为基于 `NODE_BLUEPRINTS` 配置字典的策略模式；将 $O(N^2)$ 的子孙节点查找算法优化为基于邻接表的 BFS ($O(N)$) 广度优先搜索；并对文件内部进行逻辑分区（基础工具、快照比对、几何计算、树遍历、节点工厂）。

**优势**：节点组件拆分后，单个节点的状态更新不会导致其他无关节点组件的闭包重建，显著提升了大型画布的渲染性能；工具函数的重构消除了硬编码，提升了代码的可扩展性与执行效率。

### 4. 多模型与算力调度
- **AMD 生态兼容**：框架不依赖任何 NVIDIA 专属生态（如 CUDA 强绑定），Ollama 侧车支持 AMD ROCm 或纯 CPU 推理，云端模型通过标准 OpenAI 兼容 API 调用。
- **算力保护**：Python Sidecar (`main.py`) 通过 FastAPI 的 `Request.is_disconnected()` 实时检测前端 SSE 连接状态。若用户点击“停止生成”或关闭应用，Sidecar 会立即切断推理算力，避免资源浪费。
