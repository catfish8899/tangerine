tangerine 桌面 AI 助手 - 阶段性开发说明书 (流式流转解耦 & 角色系统 & 自动化工作流画布初版)
本项目是一款基于 Windows 11 Fluent Design（暗黑微光风格）的 AI 智能聊天客户端。应用后端使用 Rust (Tauri 2.0) 管理窗口及生命周期，并通过本地 Python Sidecar 服务桥接调用云端 DeepSeek/Gemini API 及本地 Ollama 服务，确保 API 密钥的安全性和本地扩展能力。

一、 技术栈组成 & 新增前端依赖
宿主与壳层 (Desktop Shell)

Tauri 2.0：轻量级桌面开发框架。使用 Rust 构建，负责管理应用窗口、文件系统访问，以及在后台自动拉起和管理 Python Sidecar 子进程。
@tauri-apps/plugin-shell：用于在桌面壳中跨进程、安全地调用 Windows 操作系统的默认浏览器（Chrome / Edge）打开网页引用源。
安装命令： pnpm tauri add shell
前端界面 (Frontend UI)

React 18 & TypeScript：用于高效率的前端组件开发与强类型保证。
Vite 6：极其快速的现代化前端构建与热重载工具。
Tailwind CSS v4 (@tailwindcss/postcss)：采用全新 v4 架构。通过引入 @import "tailwindcss"; 结合 @source 动态规则，极速提取 Fluent 风格组件样式。
Lucide React：现代、简洁、一致的图标库。
@xyflow/react：React Flow 新版图编排核心库，用于自动化工作流画布、节点拖拽生成、节点连线、缩放/平移以及集合区域 NodeResizer 可调尺寸能力。
前端渲染与高亮依赖
为了实现高可读性的 AI 消息回复、代码框样式与高亮：

react-markdown: 用于解析并渲染 AI 返回的 Markdown 数据。
remark-gfm: 插件，支持 GitHub 风格的 Markdown（表格、任务列表等）。
rehype-highlight & highlight.js: 插件及主题包，用于解析代码块并提供 Atom One Dark 彩色高亮样式。
安装命令：
pnpm add react-markdown remark-gfm rehype-highlight highlight.js

自动化工作流画布依赖安装命令：
pnpm add @xyflow/react

二、 核心功能与架构升级说明
1. SSE (Server-Sent Events) 流式传输架构：
   后端 FastAPI 路由 `/chat` 统一改造为返回 `StreamingResponse` (基于 `text/event-stream` 协议)。
   前端优化了原生 `fetch` 流式数据解析器，加入了高度兼容的 `extractTextDelta` 提取管道，支持 `choices.delta.content`、自定义 `text` 字段、`message.content` 等多格式多端混吐解析，解决打字机不出字或幽灵空框的痛点。

2. 角色系统与提示词隔离设计（新增）：
   - 磨砂半透明控制面板：采用 Fluent Design 的 `backdrop-blur-md` 磨砂高光风格，铺满软件界面。支持卡片化编辑、创建与物理删除自定义系统提示词。
   - 唯一性重名校验：在卡片编辑保存时，启动多轮不重名断言，从前端规避同名角色数据污染。
   - 对话生命周期绑定：遵循“只能在空白会话中选择角色”的严格交互规则。对话一旦产生内容，选择器即时隐藏；已选中的角色卡片会以微光 System 气泡常驻展示在消息视窗最顶部，以便用户随时掌控当前 AI 的心智状态。
   - 提示词安全解耦：仅将会话绑定的自定义系统提示词通过 API 的 `system` 消息通道送达大模型。联网、工具组装等临时提示词绝不污染展示区的角色 System 气泡，保证展示纯粹性。

3. 本地 Ollama 通道高抗错改造：
   为彻底解决本地模型偶尔不吐字及多电脑环境下连接不稳定的问题，后端 Ollama 流式链路升级为直接使用标准的 `OpenAI` SDK 直连本地 `11434` 端口的 `/v1` 兼容网关，抹平了底层 SSE 字符缓冲差异，实现了 100% 稳定的流式输出。

4. 熔断防御机制（消灭幽灵消息）：
   前端专门建立了流式熔断控制（Try-Catch Stream Breaker）。当云端接口、网络环境或本地 Ollama 服务在流式建立及握手阶段抛出异常时，前端立即熔断并在内存及 LocalStorage 中彻底“抹杀”刚才为了等待流式而创建的空临时 AI 占位消息，并在下方追加带有清晰提示的 `system_err` 系统报错气泡。这完美保护了 `[User-AI]` 的严格交替对话逻辑。

5. 现代化微光交互语言：
   - 思考状态自动渐隐：点击发送后仅显示“模型正在思考中...”提示；一旦模型吐出第一个字符或展现联网检索轨迹（出字瞬间），提示气泡立即消失，避免空间重叠与无谓抖动。
   - 气泡隐入：在回答出第一个字符之前，AI 空文本框完全隐形。
   - 属性淡入：大模型底部的提供商、模型名称、Token 消耗、响应时间等系统专业 Metadata 只有在整个流式加载完成（isLoading 结束）后，才会整齐、优雅地淡入显示。

6. 自动化工作流画布初版（新增）：
   - 独立会话类型：在 ChatSession 中新增 type 字段，用于区分普通聊天会话与自动化流程会话。普通聊天为 "chat"，自动化流程为 "automation"。
   - 左侧统一会话列：点击“新自动化流程”后，会在左侧会话列表中创建一条新的自动化流程记录。它与“新对话”创建的普通聊天记录同处于左侧竖列，但右侧展开内容完全不同。
   - 右侧独立画布：自动化流程会话右侧不再展示聊天输入框与消息流，而是展示基于 @xyflow/react 的深色点阵画布。
   - 底部工作流菜单栏：画布底部提供悬浮磨砂菜单栏，包含“角色”、“工具”、“开关”、“集合”四类上拉菜单。
     - “角色”菜单：从本地角色系统 localStorage 数据 tangerine_roles 中读取角色列表，可将具体角色拖入画布生成角色节点。
     - “工具”菜单：当前内置“联网搜索”工具节点，后续可扩展更多工具节点。
     - “开关”菜单：当前内置“开始对话”入口节点，后续用于触发自动化流程执行。
     - “集合”菜单：当前内置“集合区域”节点，支持生成可调大小的画布范围框。
   - 拖拽实现策略：由于 Tauri WebView + React Flow 场景下浏览器原生 HTML5 draggable / dataTransfer / drop 机制存在不稳定情况，曾出现拖拽到画布显示“禁止”图标的问题，因此当前工作流菜单采用 Pointer 事件模拟拖拽：
     - pointerdown：从菜单项开始拖拽；
     - pointermove：显示跟随鼠标的幽灵预览；
     - pointerup：若鼠标释放位置位于画布区域内，则通过 screenToFlowPosition 转换坐标并创建 React Flow 节点。
   - 自定义节点初版：
     - automationRoleNode：角色节点，展示角色名称与系统提示词摘要。
     - automationToolNode：工具节点，当前用于联网搜索工具。
     - automationTriggerNode：开关/入口节点，当前用于“开始对话”。
     - automationCollectionNode：集合区域节点，使用 NodeResizer 支持选中后调整区域大小。
   - 暗黑视觉适配：隐藏 React Flow 右下角默认 attribution；对 Controls 控制栏进行暗黑主题样式覆盖；节点、连线、拖拽预览均采用 Windows 11 暗黑微光风格。
   - 当前边界：本阶段只完成工作流画布的初版交互与节点生成。尚未实现工作流数据持久化、自动执行引擎、节点配置面板、集合区域父子归属检测、节点删除/复制粘贴快捷键、流程合法性校验等高级功能。

四、 项目环境依赖与多电脑协同开发
为了能在不同的电脑上通过 GitHub 克隆代码后快速恢复开发，请确保目标电脑满足以下环境：

1. 全局系统环境要求

Node.js：建议安装 v18.x 或 v20.x LTS 版本。
pnpm：推荐使用的前端包管理器。
Rust 编译链：安装 Rustup，且 Windows 电脑需要 C++ 生成工具（VS C++ 桌面开发工作负载）。
Python 3：确保 python 已添加到系统环境变量 PATH 中。
环境变量：
  - `DEEPSEEK_API_KEY`：云端 DeepSeek 推理秘钥。
  - `GEMINI_API_KEY`：云端 Gemini 推理密钥。
  - `TAVILY_API_KEY`：Tavily 联网搜索 API 密钥。
本地 service（可选）：若要启用离线 AI，建议安装本地 Ollama 并拉取相应模型。

2. 跨电脑克隆后的快速启动步骤
在新电脑上克隆仓库后，只需在项目根目录下依次执行以下命令：

步骤 A：安装前端及 Tauri 依赖
pnpm install
pnpm tauri add shell

若当前分支包含自动化工作流画布功能，请确保安装 React Flow 新版图编排依赖：
pnpm add @xyflow/react

步骤 B：安装 Python Sidecar 依赖
pip install -r src-tauri/py_sidecar/requirements.txt

步骤 C：清理缓存并一键启动开发预览
如遇到样式未能实时加载等现象，建议清理 Vite 缓存：
PowerShell: Remove-Item -Recurse -Force .vite
启动开发服务：pnpm tauri dev

步骤 D（可选调试）：启动开发者实时调试面板
在另一个终端下进入 `src-tauri/py_sidecar/`，运行：
python debugger.py
即可开启硬件级时序上下文监控面板。

五、 自动化工作流画布开发注意事项

1. 拖拽机制说明
当前自动化工作流底部菜单没有使用浏览器原生 draggable / dataTransfer / drop 机制。原因是在 Windows 11 Tauri WebView + @xyflow/react 的组合中，原生拖拽存在兼容性问题，曾出现拖拽到画布时显示“禁止”图标且无法触发 onDrop 的现象。

当前采用 Pointer 事件模拟拖拽：
- 菜单项 onPointerDown 开始拖拽；
- window pointermove 更新幽灵预览位置；
- window pointerup 判断释放点是否在画布 wrapper 内；
- 若释放点有效，则调用 screenToFlowPosition 将屏幕坐标转换为 React Flow 世界坐标；
- 最后 setNodes 创建节点。

后续若继续扩展底部菜单或节点模板，应优先沿用该 Pointer 拖拽模式，避免回退到原生 HTML5 拖拽。

2. React Flow 类型约束说明
@xyflow/react 新版中 Node<TData> 的 TData 需要满足 Record<string, unknown>。因此自动化节点数据类型 AutomationNodeData 必须显式继承 Record<string, unknown>，否则 TypeScript 会报错。

当前核心类型设计：
- AutomationNodeKind = "role" | "tool" | "trigger" | "collection"
- AutomationTemplate：底部菜单模板项
- AutomationNodeData extends Record<string, unknown>：画布节点数据
- AutomationFlowNode = Node<AutomationNodeData>
- AutomationFlowEdge = Edge

ReactFlow 组件也应同步声明泛型：
ReactFlow<AutomationFlowNode, AutomationFlowEdge>

3. 当前节点类型
当前注册的自动化节点类型：
- automationRoleNode：角色节点
- automationToolNode：工具节点
- automationTriggerNode：开关/入口节点
- automationCollectionNode：集合范围节点

集合节点使用 NodeResizer，选中后可调整大小。

4. 角色菜单数据来源
自动化流程的“角色”菜单直接读取 localStorage 中的 tangerine_roles，与现有 RolesModal 角色系统共用数据源。
如果没有角色，则显示“暂无角色”的禁用菜单项。

5. 底部上拉菜单交互说明
自动化工作流底部菜单支持鼠标悬停展开细分选项。由于菜单项与上拉面板之间存在视觉间距，曾出现鼠标移出底部按钮后上拉面板立刻关闭、导致无法移动到细分选项的问题。

当前处理策略：
- 鼠标离开菜单容器后延迟关闭；
- 在底部按钮与上拉面板之间增加透明连接桥；
- 保证用户可以从底部按钮自然移动到上方细分选项并开始拖拽。

6. 当前未完成能力
当前只是工作流画布和节点模板的初版，不包含完整自动化执行能力。后续建议按以下顺序继续开发：
- 画布数据持久化：按 automation session id 保存 nodes / edges；
- 节点右键菜单：删除、复制、查看配置；
- 节点配置面板：角色节点可切换角色，工具节点可配置参数，开关节点可配置触发方式；
- 集合区域父子归属：拖入集合框后自动设置 parentId，拖出后解除；
- 工作流执行引擎：从“开始对话”节点出发，根据连线拓扑解析角色、工具、模型调用顺序；
- 流程合法性校验：避免孤立入口、多入口冲突、环形依赖、缺少角色节点等；
- 自动化流程与普通聊天的执行桥接：运行工作流后可将结果写入聊天消息流或创建独立执行日志。
