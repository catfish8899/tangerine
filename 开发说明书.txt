### tangerine 桌面 AI 助手 - 阶段性开发说明书 (流式流转解耦 & 角色系统 & 自动化工作流画布模块化解耦版)

本项目是一款基于 Windows 11 Fluent Design（暗黑微光风格）的 AI 智能聊天客户端。应用后端使用 Rust (Tauri 2.0) 管理窗口及生命周期，并通过本地 Python Sidecar 服务桥接调用云端 DeepSeek/Gemini API 及本地 Ollama 服务，确保 API 密钥的安全性和本地扩展能力。

一、 技术栈组成 & 新增前端依赖
宿主与壳层 (Desktop Shell)

Tauri 2.0：轻量级桌面开发框架。使用 Rust 构建，负责管理应用窗口、文件系统访问，以及在后台自动拉起和管理 Python Sidecar 子进程。
@tauri-apps/plugin-shell：用于在桌面壳中跨进程、安全地调用 Windows 操作系统的默认浏览器（Chrome / Edge）打开网页引用源。
安装命令： pnpm tauri add shell
前端界面 (Frontend UI)

React 18 & TypeScript：用于高效率的前端组件开发与强类型保证。
Vite 6：极其快速的现代前端构建与热重载工具。
Tailwind CSS v4 (@tailwindcss/postcss)：采用全新 v4 架构。通过引入 @import "tailwindcss"; 结合 @source 动态规则，极速提取 Fluent 风格组件样式。
Lucide React：现代、简洁、一致的图标库。
@xyflow/react：React Flow 新版图编排核心库，用于自动化工作流画布、节点拖拽生成、节点连线、缩放/平移以及集合区域 NodeResizer 可调尺寸能力。
前端渲染与高亮依赖
为了实现高可读性的 AI 消息回复、代码框样式与高亮：

react-markdown: 用于解析并渲染 AI 返回的 Markdown 数据。
remark-gfm: 插件，支持 GitHub 风格的 Markdown（表格、任务列表等）。
rehype-highlight & highlight.js: 插件及主题包，用于解析代码块并提供 Atom One Dark 彩色高亮样式。
安装命令：
pnpm add react-markdown remark-gfm rehype-highlight highlight.js

自动化工作流画布依赖安装命令：
pnpm add @xyflow/react

二、 核心功能与架构升级说明
1. SSE (Server-Sent Events) 流式传输架构：
   后端 FastAPI 路由 `/chat` 统一改造为返回 `StreamingResponse` (基于 `text/event-stream` 协议)。
   前端优化了原生 `fetch` 流式数据解析器，加入了高度兼容的 `extractTextDelta` 提取管道，支持 `choices.delta.content`、自定义 `text` 字段、`message.content` 等多格式多端混吐解析，解决打字机不出字或幽灵空框的痛点。

2. 角色系统与提示词隔离设计（新增）：
   - 磨砂半透明控制面板：采用 Fluent Design 的 `backdrop-blur-md` 磨砂高光风格，铺满软件界面。支持卡片化编辑、创建与物理删除自定义系统提示词。
   - 唯一性重名校验：在卡片编辑保存时，启动多轮不重名断言，从前端规避同名角色数据污染。
   - 对话生命周期绑定：遵循“只能在空白会话中选择角色”的严格交互规则。对话一旦产生内容，选择器即时隐藏；已选中的角色卡片会以微光 System 气泡常驻展示在消息视窗最顶部，以便用户随时掌控当前 AI 的心智状态。
   - 提示词安全解耦：仅将会话绑定的自定义系统提示词通过 API 的 `system` 消息通道送达大模型。联网、工具组装等临时提示词绝不污染展示区的角色 System 气泡，保证展示纯粹性。

3. 本地 Ollama 通道高抗错改造：
   为彻底解决本地模型偶尔不吐字及多电脑环境下连接不稳定的问题，后端 Ollama 流式链路升级为直接使用标准的 `OpenAI` SDK 直连本地 `11434` 端口的 `/v1` 兼容网关，抹平了底层 SSE 字符缓冲差异，实现了 100% 稳定的流式输出。

4. 熔断防御机制（消灭幽灵消息）：
   前端专门建立了流式熔断控制（Try-Catch Stream Breaker）。当云端接口、网络环境或本地 Ollama 服务在流式建立及握手阶段抛出异常时，前端立即熔断并在内存及 LocalStorage 中彻底“抹杀”刚才为了等待流式而创建的空临时 AI 占位消息，并在下方追加带有清晰提示的 `system_err` 系统报错气泡。这完美保护了 `[User-AI]` 的严格交替对话逻辑。

5. 现代化微光交互语言：
   - 思考状态自动渐隐：点击发送后仅显示“模型正在思考中...”提示；一旦模型吐出第一个字符或展现联网检索轨迹（出字瞬间），提示气泡立即消失，避免空间重叠与无谓抖动。
   - 气泡隐入：在回答出第一个字符之前，AI 空文本框完全隐形。
   - 属性淡入：大模型底部的提供商、模型名称、Token 消耗、响应时间等系统专业 Metadata 只有在整个流式加载完成（isLoading 结束）后，才会整齐、优雅地淡入显示。

6. 自动化工作流画布（深度更新、增强与模块化解耦）：
   - 独立会话类型：在 ChatSession 中新增 type 字段，用于区分普通聊天会话与自动化流程会话。普通聊天为 "chat"，自动化流程为 "automation"。
   - 跨会话画布物理隔离：每个自动化流程会话拥有其专属的 `sessionId`。画布数据持久化文件统一命名为 `tangerine_automation_flow_{sessionId}` 并实时存储至 LocalStorage，彻底解决了不同自动化画布之间数据相互覆盖、交叉污染的痛点。
   - 会话物理销毁联动：在左侧会话列表中右键彻底删除某个自动化流程会话时，系统将同步、彻底清除该会话专属的 LocalStorage 画布数据，确保无无效脏数据残留。
   - 右上角系统时间组件：画布右上角常驻高雅、半透明磨砂的系统时间组件（SystemTimeBadge），实时同步显示高精度系统时间，格式包含：“年月日、12小时制（上午/下午 时:分:秒）”。
   - 右下角暗黑主题小地图：画布右下角嵌入高度定制的 `MiniMap` 小地图组件，背景色为深黑（#181818），遮罩层呈半透明黑色。不同节点类型映射不同的小地图高亮微光颜色：集合节点高亮紫色，角色节点高亮金黄，工具节点高亮天蓝，对话节点高亮青色等，提升大型复杂工作流的全局检索操控体验。
   - 底部悬浮磨砂菜单栏：提供“角色”、“工具”、“开关”、“集合”、“对话”、“硬件”、“定时”七大类上拉菜单，均采用 Pointer 事件模拟拖拽生成。
     - “角色”菜单：读取本地 LocalStorage 数据中 `tangerine_roles` 列表，拖拽生成角色节点。
     - “工具”菜单：提供精简版“联网搜索”工具。
     - “开关”菜单：内置“开始对话”入口节点。
     - “集合”菜单：内置“集合区域”节点，利用 NodeResizer 独立调整区域大小。
     - “对话”菜单：内置“对话”节点，画布内节点带有一款高度抗冲突的输入框（`.nodrag` 区域），用户能直接在画布中双击、聚焦并录入多行文本对话。
     - “硬件”菜单：内置“硬件”节点，节点渲染并常驻“连接硬件”文字说明，用于硬件控制。
     - “定时”菜单：内置“定时”节点，带有专门的输入组件（`.nodrag` 区域），供用户快速手动录入如 “2026年01月01日 下午 3:30” 的年月日 + 12小时制时间数据。
   - 键盘快捷键组合与撤销历史机制：
     - **撤销重做 (Ctrl + Z)**：内置轻量级历史快照管理器，最大深度 80 步。支持节点拖拽、集合拉伸、节点新建、节点物理删除、连线生成以及粘贴等所有操作的一步撤销，保障画布编辑安全。
     - **徹底删除 (Delete / Backspace)**：一键删除当前选中的节点、边连线。如果删除的是集合节点，其内部已被吸附归宿的子孙节点及相关连线将合并执行联动级联删除，拒绝孤立悬空节点。
     - **深度复制与粘贴 (Ctrl + C & Ctrl + V)**：支持多选、跨画布复制粘贴。若选中的元素中包含“集合”节点，系统会自动捕获并复制该集合区域内吸附的**所有子节点以及它们之间的完整边连线**。粘贴时，以当前鼠标在画布内的世界坐标为锚点，内部子节点相对集合的物理相对坐标保持原样不变，且粘贴出的所有节点和边将自动被赋予全新、抗冲突的唯一 ID 标识，防止画布节点污染。
   - 自动化工作流画布已完成前端模块化解耦：
     - 原 `AutomationWorkspace.tsx` 不再承载全部画布业务逻辑，而是作为轻量入口组件存在。
     - 画布主编辑器逻辑被下沉至 `src/components/automation/FlowEditor.tsx`。
     - 节点类型、工具栏、拖拽幽灵预览、系统时间、存储逻辑、类型定义、节点工具函数分别拆分到独立模块。
     - 本次解耦保持原有业务逻辑不变，仅调整代码组织结构，提高后续维护性、扩展性与多人协作效率。

四、 项目环境依赖与多电脑协同开发
为了能在不同的电脑上通过 GitHub 克隆代码后快速恢复开发，请确保目标电脑满足以下环境：

1. 全局系统环境要求

Node.js：建议安装 v18.x 或 v20.x LTS 版本。
pnpm：推荐使用的前端包管理器。
Rust 编译链：安装 Rustup，且 Windows 电脑需要 C++ 生成工具（VS C++ 桌面开发工作负载）。
Python 3：确保 python 已添加到系统环境变量 PATH 中。
环境变量：
  - `DEEPSEEK_API_KEY`：云端 DeepSeek 推理秘钥。
  - `GEMINI_API_KEY`：云端 Gemini 推理密钥。
  - `TAVILY_API_KEY`：Tavily 联网搜索 API 密钥。
本地 service（可选）：若要启用离线 AI，建议安装本地 Ollama 并拉取相应模型。

2. 跨电脑克隆后的快速启动步骤
在新电脑上克隆仓库后，只需在项目根目录下依次执行以下命令：

步骤 A：安装前端及 Tauri 依赖
pnpm install
pnpm tauri add shell

若当前分支包含自动化工作流画布功能，请确保安装 React Flow 新版图编排依赖：
pnpm add @xyflow/react

步骤 B：安装 Python Sidecar 依赖
pip install -r src-tauri/py_sidecar/requirements.txt

步骤 C：清理缓存并一键启动开发预览
如遇到样式未能实时加载等现象，建议清理 Vite 缓存：
PowerShell: Remove-Item -Recurse -Force .vite
启动开发服务：pnpm tauri dev

步骤 D（可选调试）：启动开发者实时调试面板
在另一个终端下进入 `src-tauri/py_sidecar/`，运行：
python debugger.py
即可开启硬件级时序上下文监控面板。

五、 自动化工作流画布开发注意事项

1. 拖拽机制说明
当前自动化工作流底部菜单没有使用浏览器原生 draggable / dataTransfer / drop 机制。原因是在 Windows 11 Tauri WebView + @xyflow/react 的组合中，原生拖拽存在兼容性问题，曾出现拖拽到画布时显示“禁止”图标且无法触发 onDrop 的现象。

当前采用 Pointer 事件模拟拖拽：
- 菜单项 onPointerDown 开始拖拽；
- window pointermove 更新幽灵预览位置；
- window pointerup 判断释放点是否在画布 wrapper 内；
- 若释放点有效，则调用 screenToFlowPosition 将屏幕坐标转换为 React Flow 世界坐标；
- 最后 setNodes 创建节点。

当前相关逻辑分布：
- `AutomationToolbar.tsx`：负责底部菜单项的 PointerDown 入口。
- `DragGhost.tsx`：负责拖拽过程中的幽灵预览。
- `FlowEditor.tsx`：负责全局 pointermove / pointerup 监听、释放点判断、screenToFlowPosition 转换与节点创建。

后续若继续扩展底部菜单或节点模板，应优先沿用该 Pointer 拖拽模式，避免回退到原生 HTML5 拖拽。

2. React Flow 类型约束说明
@xyflow/react 新版中 Node<TData> 的 TData 需要满足 Record<string, unknown>。因此自动化节点数据类型 AutomationNodeData 必须显式继承 Record<string, unknown>，否则 TypeScript 会报错。

当前相关类型已统一收敛在：
`src/components/automation/automationTypes.ts`

在进行数组连接或重构画布选中态重置逻辑时，TypeScript 容易将 `map` 后返回的局部对象窄化推断（例如将 `selected: false` 锁死为 `boolean` 非可选类型），从而与 React Flow 宽松的 `AutomationFlowNode[]` （其中 `selected?: boolean` 为可选）发生重载调用冲突。应当避免直接使用 Array 的原生 `.concat()`，改为在返回解构数组前显式声明函数的输出类型，例如：
`setNodes((currentNodes): AutomationFlowNode[] => { ... })`
通过对数组或 `map` 转换器施加精确断言，能够彻底规避编译期重载匹配失效。

3. 节点定制类型
当前注册的自动化节点类型：
- `automationRoleNode`：角色节点。
- `automationToolNode`：工具节点（当前用于精简后的“联网搜索”，仅显示节点名称与工具名称）。
- `automationTriggerNode`：开关/入口节点（“开始对话”）。
- `automationCollectionNode`：集合范围节点（使用 NodeResizer 调整区域尺寸）。
- `automationConversationNode`：对话节点，带双向同步 textarea 文本框。
- `automationHardwareNode`：硬件节点，常驻“连接硬件”标题。
- `automationTimerNode`：定时节点，带年月日+12小时制时间格式 input 输入框。

当前节点组件与 nodeTypes 注册表已统一收敛在：
`src/components/automation/AutomationNodes.tsx`

若后续新增节点类型，需要同步修改：
- `automationTypes.ts` 中的 `AutomationNodeKind`。
- `automationTypes.ts` 中的 `AutomationTemplate["icon"]` 联合类型。
- `automationNodeUtils.ts` 中的 `getNodeSize` fallback 尺寸映射。
- `automationNodeUtils.ts` 中的 `buildAutomationNode` 节点构建逻辑。
- `AutomationNodes.tsx` 中新增节点组件与 `nodeTypes` 注册。
- `AutomationToolbar.tsx` 中新增对应模板菜单或已有菜单项。

4. 角色菜单数据来源
自动化流程的“角色”菜单直接读取 localStorage 中的 `tangerine_roles`，与现有 RolesModal 角色系统共用数据源。
如果没有角色，则显示“暂无角色”的禁用菜单项。

当前角色读取逻辑：
- `automationStorage.ts` 中的 `safeLoadRoles` 负责安全读取角色数据。
- `FlowEditor.tsx` 中的 `roleItems` 负责将 Role 数据转换为可拖拽的 AutomationTemplate。
- `AutomationToolbar.tsx` 负责渲染“角色”上拉菜单。

5. 底部上拉菜单交互说明
自动化工作流底部菜单支持鼠标悬停展开细分选项。由于菜单项与上拉面板之间存在视觉间距，曾出现鼠标移出底部按钮后上拉面板立刻关闭、导致无法移动到细分选项的问题。

当前处理策略：
- 鼠标离开菜单容器后延迟关闭；
- 在底部按钮与上拉面板之间增加透明连接桥；
- 保证用户可以从底部按钮自然移动到上方细分选项并开始拖拽。

当前相关实现位于：
`src/components/automation/AutomationToolbar.tsx`

6. 键盘快捷键与撤销历史机制
   - 画布内的快捷键监听需要特别注意**输入框焦点防御**。在事件回调头部，必须通过调用 `isEditableElement(event.target)` 来对正在进行文本编辑（如对话、定时节点中的 input, textarea 或是其他 contenteditable 区域）的用户输入行为进行豁免拦截，避免在输入字符时意外触发删除节点、复制、粘贴或撤销动作。
   - `Ctrl+C` 深度复制集合时，算法会自动利用广度/深度优先逻辑（通过 `getDescendantNodeIds`）递归扫描集合节点内吸附的每一级子孙节点，合并收集处于两点之间的所有边缘连线（edges），并将它们的绝对世界坐标一并导出。
   - `Ctrl+V` 深度粘贴时，系统以鼠标当前的流坐标（flow coordinates）为贴图中心点，把剪贴板内所有节点的旧绝对坐标对准新中心，重新计算出它们的位移差（delta offset），以此算出粘贴后的新绝对坐标，对于包含集合子级的节点，根据重新计算后的相对父节点坐标进行再转换，使其能够天衣无缝地维持在粘贴后的新集合区域内，不发生错位或散架。
   - 撤销机制利用纯函数 `cloneFlowSnapshot` 对 `nodes` 和 `edges` 状态深度深拷贝。每次在节点改变大小（维度改变）或拖拽节点开始时抓取初始镜像，当有实质性操作发生时，向快照队列推进新记录。

当前相关逻辑分布：
- `FlowEditor.tsx`：快捷键监听、删除、复制、粘贴、撤销、历史栈推进。
- `automationNodeUtils.ts`：`isEditableElement`、`cloneFlowSnapshot`、`areSnapshotsEqual`、`getDescendantNodeIds`、`getNodeAbsolutePosition`、`getNodeSize` 等纯工具函数。
- `automationStorage.ts`：自动化剪贴板 LocalStorage 读写。
- `automationConstants.ts`：历史记录最大深度 `MAX_HISTORY_LENGTH`。

7. 自动化工作流模块化维护说明
   当前自动化工作流画布已经从单一大文件 `AutomationWorkspace.tsx` 解耦为多个职责明确的小模块。后续维护时应遵循以下边界：

   - `AutomationWorkspace.tsx`
     仅作为自动化画布入口组件，负责外层容器和 `ReactFlowProvider` 注入。除非入口层职责变化，否则不应再向该文件堆叠业务逻辑。

   - `FlowEditor.tsx`
     负责 React Flow 主画布状态与核心交互编排，包括：
     - nodes / edges 状态；
     - 持久化加载与保存；
     - Pointer 拖拽生成节点；
     - 节点拖拽与集合吸附；
     - 连线生成；
     - 快捷键监听；
     - 删除、复制、粘贴、撤销；
     - MiniMap、Controls、Background 等画布级组件挂载。

   - `AutomationNodes.tsx`
     只放置节点渲染相关逻辑，包括节点外观、节点内部 input/textarea 数据同步、NodeResizer 和 nodeTypes 注册。不要在该文件中加入全局快捷键、画布持久化等主画布逻辑。

   - `AutomationToolbar.tsx`
     只放置底部悬浮菜单和模板列表相关逻辑。后续新增菜单分类或菜单项时优先修改这里。

   - `DragGhost.tsx`
     只负责拖拽幽灵预览，不负责节点创建。

   - `SystemTimeBadge.tsx`
     只负责系统时间展示。

   - `automationTypes.ts`
     只维护类型定义，尤其是节点数据结构、模板结构、快照结构、剪贴板结构。

   - `automationStorage.ts`
     只维护 LocalStorage 读写与容错解析，不放 UI 逻辑。

   - `automationNodeUtils.ts`
     只维护与节点、坐标、快照、集合吸附、节点构建相关的纯函数或近似纯函数。

   该拆分原则的目标是：确保后续新增工作流执行引擎、节点属性面板、流程校验等能力时，不再让入口组件重新膨胀为超大文件。

8. 当前未完成能力（已更新）
   在完成了多会话隔离、撤销重做历史、快捷键、深度复制粘贴、高精度时钟、暗黑 MiniMap、输入型节点定制以及自动化画布前端模块化解耦后，接下来的迭代开发建议按以下方向进行：
   - **工作流执行引擎**：从“开始对话”等 trigger 节点出发，根据有向无环连线拓扑关系，在 Python Sidecar 侧建立运行管道，解析角色、工具和对话输入节点的调用顺序。
   - **节点参数细化面板**：双击节点或右键菜单中加入“属性配置”抽屉，供用户为工具节点、对话节点微调更深层次的参数模型设定（如：更换大模型提供商、设置延时长度等）。
   - **流程连线合法性校验**：自动侦测画布内的孤立节点、环形死循环连线、多触发器冲突等异常状态并进行红色高亮警告。
